# Feature Specification: Tesseract Client UX Loop

**Feature Branch**: `010-tesseract-ux-loop`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "系统化优化客户端用户交互体验，包括硬件状态位置、命名统一、数字孪生更新节流、对话 markdown/diagram 渲染、确认工作流后直接显示到 n8n、去掉中间表格、Studio/数字孪生层级、Skills 库卡片、热插拔阶段图片上传与单选多选配置渲染、对话模式下基于当前硬件可运行的 Skills GUI 卡片。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 对话构建闭环直接落到工作区 (Priority: P1)

作为正在通过聊天搭建硬件工作流的用户，我希望 AI 的需求总结、流程图、确认工作流和 n8n 工作区显示是一条连续路径，不需要再补点“创建工作流”或面对冗余表格。

**Why this priority**: 这是当前教学/对话模式的主闭环。如果这里断开，用户会认为系统没有真正生成工作流。

**Independent Test**: 在空工作区中发起一次工作流构建请求，查看 AI 输出的 markdown 总结和可视化流程图，点击“确认工作流”后应直接在当前 n8n 工作区显示对应工作流，无需额外点击“创建工作流”。

**Acceptance Scenarios**:

1. **Given** 用户在聊天中完成需求确认，**When** AI 输出工作流确认卡片，**Then** 页面必须显示格式化后的总结文本和流程图，而不是原始表格块与未格式化长文本。
2. **Given** 当前工作区还没有显示任何工作流，**When** 用户点击“确认工作流”，**Then** 当前工作区必须自动切换并显示新生成的工作流，且不再要求用户额外点击“创建工作流”。
3. **Given** 工作流已成功显示到当前工作区，**When** 用户继续查看聊天记录，**Then** 聊天中的按钮文案与状态必须反映“已推送到工作区”，而不是继续显示要求重复创建的动作。

---

### User Story 2 - 对话模式按真实 Skills 与硬件状态分流 (Priority: P1)

作为想直接运行现有技能的用户，我希望对话模式能根据现有 Skills 库和当前心跳回来的硬件状态给出可点击技能卡片，并在点击后自动决定是直接运行还是进入热插拔/配置步骤。

**Why this priority**: 这是从“聊天理解”走向“真实技能运行”的入口，也是 specs001 定义的 A/B/C 分支在桌面端的主要体现。

**Independent Test**: 在对话模式下让 AI 匹配到一个现有技能，页面必须显示带硬件就绪状态的技能卡片；点击卡片后，如果硬件齐全则直接显示并加载工作流，如果硬件不齐则进入热插拔/配置步骤。

**Acceptance Scenarios**:

1. **Given** Skills 库中存在与用户意图匹配的技能，**When** 用户进入对话模式并提出该需求，**Then** 页面必须显示技能卡片、所需硬件和当前硬件是否就绪。
2. **Given** 技能所需硬件全部在线，**When** 用户点击技能卡片，**Then** 系统必须直接显示该技能的工作流并进入运行/下发步骤。
3. **Given** 技能所需硬件不完整，**When** 用户点击技能卡片，**Then** 系统必须进入热插拔/配置流程，并明确指出缺失项。
4. **Given** 用户输入的需求在技能库中不存在，**When** AI 完成语义判断，**Then** 页面必须回到教学模式/闲聊分流，而不是继续显示伪造的现成技能卡片。

---

### User Story 3 - 热插拔与配置步骤可在聊天中完整完成 (Priority: P1)

作为正在补齐硬件和配置参数的用户，我希望热插拔阶段的图片上传、单选、多选、文字输入等配置项都能在聊天界面里被正确渲染和提交，而不是退回外部工具或手工操作。

**Why this priority**: 这是工作流从“结构正确”走到“参数可运行”的关键缺口。没有它，用户仍然无法在一个界面内完成配置闭环。

**Independent Test**: 让 Agent 返回一组混合配置项，包括人脸图片上传、单选和多选；用户必须能在聊天界面完成配置并推进到“所有节点配置完毕”。

**Acceptance Scenarios**:

1. **Given** Agent 要求人脸识别节点上传目标人脸图片，**When** 聊天界面渲染该步骤，**Then** 用户必须能在聊天中选择目标人物并上传图片。
2. **Given** Agent 返回单选、多选或文本类配置项，**When** 聊天界面显示该步骤，**Then** 每种控件都必须以对应的 GUI 形态渲染并能提交。
3. **Given** 所有配置项都已完成，**When** 用户查看步骤状态，**Then** 页面必须明确显示“所有节点配置完毕”并给出后续动作按钮。

---

### User Story 4 - 桌面壳层级、命名与库卡片统一 (Priority: P2)

作为桌面端用户，我希望客户端品牌命名、硬件状态入口、数字孪生窗口层级和 Skills 库卡片都统一且清晰，不会互相遮挡或用旧命名混淆。

**Why this priority**: 这直接影响感知质量和调试效率，但不阻断工作流主闭环，因此优先级低于前 3 个用户路径。

**Independent Test**: 启动桌面端后检查顶部品牌、左侧硬件状态入口、数字孪生和 Studio 的遮挡关系，以及 Skills 库卡片样式与内容是否满足要求。

**Acceptance Scenarios**:

1. **Given** 用户启动桌面端，**When** 查看顶栏和聊天区角色名，**Then** 品牌文案必须显示为 “Tesseract” 和 “Tess”，不再出现旧命名。
2. **Given** 用户打开数字孪生和 Studio/Skills 库，**When** 多窗口同时存在，**Then** Studio 必须保持在数字孪生之上，避免被其遮挡。
3. **Given** Skills 库中存在技能条目，**When** 用户查看卡片，**Then** 每张卡片必须展示工作流名称、简介、所需硬件、追溯按钮和发布按钮，并且绑定对应工作流 JSON。

### Edge Cases

- 当 MQTT 心跳持续到达但组件集合没有变化时，数字孪生不得反复抖动刷新。
- 当聊天界面收到无法渲染成图表的流程摘要时，系统必须给出明确的降级显示，而不是裸露错误字符串。
- 当“确认工作流”成功但当前工作区尚未准备好时，系统必须自动补齐同步，而不是丢失本次显示动作。
- 当配置步骤包含图片上传但用户未选择文件时，系统必须阻止提交并给出明确缺失提示。
- 当技能库条目缺失绑定的工作流 JSON 或追溯会话时，前端必须显示明确不可用状态，而不是展示可点击但失败的按钮。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须将桌面端左侧顶部的硬件连接状态入口移动到左侧主区域，并保持其状态可见。
- **FR-002**: 系统必须将桌面端品牌文案统一为 “Tesseract”，并将聊天中的 AI 角色名称统一为 “Tess”。
- **FR-003**: 系统必须仅在硬件组件集合、接口映射或组件状态发生变化时更新数字孪生窗口，不得因重复心跳导致可见抖动。
- **FR-004**: 系统必须将 AI 输出的需求总结文本按现有对话 markdown 渲染能力格式化显示，而不是展示原始长段纯文本。
- **FR-005**: 系统必须在“确认工作流”阶段显示可视化流程图，并移除当前在该阶段暴露给用户的表格型中间表示。
- **FR-006**: 用户点击“确认工作流”后，系统必须直接将工作流推送到当前工作区并显示在 n8n 视图中，不得再要求额外点击“创建工作流”。
- **FR-007**: 系统必须确保当前工作区与聊天中的工作流动作共享同一工作流真相源，避免聊天状态和工作区状态分裂。
- **FR-008**: 系统必须在对话模式中根据现有 Skills 库和当前硬件状态渲染可点击的技能卡片，不得继续依赖旧的 mock 技能列表或硬编码选项。
- **FR-009**: 系统必须在技能卡片中展示技能名称、简介、所需硬件与当前硬件是否就绪，并允许用户点击进入后续闭环。
- **FR-010**: 当技能所需硬件齐全时，点击技能卡片必须直接显示并加载对应工作流；当硬件不齐时，必须进入热插拔/配置步骤。
- **FR-011**: 系统必须在 Skills 库中为每个技能保存绑定的工作流 JSON，且该数据无需在卡片上展示也必须可被后续动作复用。
- **FR-012**: Skills 库卡片必须提供“回溯教学流程”动作，点击后能够跳转到对应聊天会话。
- **FR-013**: 热插拔/配置阶段必须能够在聊天界面渲染图片上传控件，并支持用户为目标人物上传图片。
- **FR-014**: 热插拔/配置阶段必须正确渲染单选、多选与文本配置项，并允许用户在聊天界面直接提交。
- **FR-015**: 当所有节点配置完毕时，系统必须给出将工作流下发到工作区/设备的明确后续动作，而不是让用户停在配置完成状态。
- **FR-016**: OpenClaw Studio 与 Skills 库界面在打开时必须位于数字孪生窗口之上，避免被数字孪生遮挡。

### Key Entities *(include if feature involves data)*

- **Workspace Workflow Target**: 当前聊天动作和工作区视图共同引用的目标工作流，决定工作区应展示哪一个流程。
- **Skill Summary Card**: 对话模式和技能库共用的技能展示单元，包含名称、简介、硬件需求、可执行状态和绑定动作。
- **Skill Record**: 本地存储的技能条目，包含技能元数据、所需硬件、绑定工作流 JSON、追溯会话标识和可发布状态。
- **Hardware Availability Snapshot**: 由心跳与运行时状态归一后的当前硬件快照，用于判断技能是否可直接运行以及数字孪生是否需要更新。
- **Dialogue Configuration Prompt**: Agent 返回给聊天界面的配置步骤单元，可能要求图片上传、单选、多选或文本输入。
- **Workflow Diagram Summary**: 在确认工作流阶段展示给用户的可视化流程摘要，用于替代表格型中间表示。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 的工作流确认操作在一次点击“确认工作流”后即可在当前工作区中显示目标工作流，无需额外补点步骤。
- **SC-002**: 在连续心跳输入但硬件集合未变化的情况下，数字孪生窗口在一次 30 秒观察中不出现可见抖动刷新。
- **SC-003**: 对话模式下，80% 的现有技能请求能够在一次匹配后显示真实技能卡片及其硬件就绪状态。
- **SC-004**: 对于包含图片上传、单选和多选的配置步骤，用户可在单一聊天界面内完成全部配置的成功率达到 95%。
- **SC-005**: Skills 库卡片加载后，用户在 10 秒内能够识别技能名称、所需硬件和可执行状态，并找到追溯入口。

## Assumptions

- 当前桌面端仍以内嵌工作区和聊天面板作为主交互载体，不新增独立编辑器产品。
- 现有技能库、会话记录和工作流快照会继续作为本地真相源的一部分被复用。
- “发布到广场”在本轮的最低可交付要求是具备稳定卡片动作和绑定数据，不要求同时完成新的外部发布系统。
- 数字孪生、工作区与聊天仍允许通过现有运行时桥接通信，但用户看到的状态必须统一。
- 现有 Agent API 已具备表达配置步骤和工作流摘要的能力，本轮可以在不破坏现有对话链的前提下调整其输出形态。
