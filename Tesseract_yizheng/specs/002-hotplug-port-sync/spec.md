# Feature Specification: Mock 插拔端口同步与数字孪生实时挂载

**Feature Branch**: `002-hotplug-port-sync`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: “mock 硬件插拔时需要把接口参数回传给 frontend，让数字孪生在插拔过程中实时更新组件位置状态；未插拔前点击数字孪生只显示底座模型 `device-001 / 5.glb`。”

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时挂载到正确接口 (Priority: P1)

作为正在测试 mock 硬件插拔的用户，我希望在插入组件时数字孪生立刻把组件挂到正确接口，而不是悬空、错位或退化到默认接口，这样我才能确认当前物理连接状态是否正确。

**Why this priority**: 这是本特性的核心价值。如果接口身份没有被保留，数字孪生就失去“实时映射物理世界”的意义。

**Independent Test**: 打开数字孪生后触发一次 mock 插入事件，事件中明确包含端口身份；无需进入完整工作流，也能独立验证组件是否实时挂到正确接口。

**Acceptance Scenarios**:

1. **Given** 用户刚打开数字孪生且尚未插入任何组件，**When** 页面完成初始化，**Then** 画面只显示底座模型 `device-001 / 5.glb`
2. **Given** 数字孪生当前只显示底座，**When** 用户触发一次带有明确端口身份的 mock 插入，**Then** 对应组件必须实时出现在该端口的正确位置
3. **Given** 用户连续插入多个组件到不同接口，**When** 各次 mock 插入事件依次到达，**Then** 数字孪生必须保留所有已连接组件，并按各自接口实时显示

---

### User Story 2 - 拔出与异常后的正确回退 (Priority: P2)

作为正在反复调试插拔的用户，我希望在组件被拔出、换口或接口信息缺失时，数字孪生能立即撤销或保守回退显示，而不是保留上一帧残影。

**Why this priority**: 只有“插得上”没有“退得干净”，数字孪生就会变成不可信的假状态面板。

**Independent Test**: 在已有挂载组件的情况下，分别触发 mock 拔出、错误端口、缺失端口三种事件，观察数字孪生是否立即回退到可信状态。

**Acceptance Scenarios**:

1. **Given** 某组件已显示在数字孪生中，**When** 用户触发对应组件的 mock 拔出，**Then** 该组件必须立即从数字孪生中移除
2. **Given** 某技能仍有其他已连接组件，**When** 单个组件被拔出，**Then** 数字孪生必须保留其他组件，只移除被拔出的组件
3. **Given** 事件中缺失有效接口身份，**When** 系统无法判断该组件应挂到哪个端口，**Then** 数字孪生不得伪装成“正确挂载完成”，而应回退到可信的最小显示状态

---

### User Story 3 - 对话/配置链路共享同一份场景真相 (Priority: P3)

作为正在通过对话模式和 mock 插拔共同调试系统的用户，我希望无论事件来自对话校验链还是 mock 快照链，数字孪生都只服从同一份场景真相，不会出现聊天卡片说“已连接”但数字孪生没更新的分裂状态。

**Why this priority**: 该特性跨 backend、容器前端和数字孪生页面；如果没有单一真相源，不同窗口会各说各话。

**Independent Test**: 分别通过对话模式校验和 mock 快照事件驱动数字孪生，确认两条路径都会得到一致的场景结果。

**Acceptance Scenarios**:

1. **Given** 对话模式正在等待用户插入硬件，**When** mock 插入事件到达并通过校验，**Then** 对话状态与数字孪生都必须更新为同一份连接结果
2. **Given** backend 已生成新的数字孪生场景，**When** 容器把场景转发给数字孪生页面，**Then** 页面不得再自行猜测挂载关系

---

### Edge Cases

- 同一组件在极短时间内连续收到“插入 -> 拔出 -> 再插入”事件时，系统必须以后到事件为准
- 多组件快照到达时，系统必须按整份快照重建当前连接视图，而不是只消费其中第一项
- 事件里出现未知组件类型但端口合法时，系统可以忽略该组件显示，但不能污染其他已知组件
- 事件里出现未知端口时，系统必须保持底座或当前可信组件，不得把组件吸附到错误端口

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在数字孪生首次打开且尚无任何已连接组件时，仅显示底座模型 `device-001 / 5.glb`
- **FR-002**: 系统 MUST 在 mock 插拔事件中保留并传递组件的接口身份，使数字孪生能够把组件挂载到正确接口
- **FR-003**: 系统 MUST 支持全量硬件快照事件，并以整份快照重建当前连接状态，而不是只消费单个组件
- **FR-004**: 系统 MUST 在 mock 拔出事件到达后立即从数字孪生中移除对应组件
- **FR-005**: 系统 MUST 在端口信息缺失、非法或不可判定时，回退到可信显示状态，而不是伪造正确挂载
- **FR-006**: 系统 MUST 让对话模式硬件校验链和 mock 快照链共享同一份数字孪生场景真相
- **FR-007**: 系统 MUST 保证 backend 输出的数字孪生场景是前端显示的唯一业务真相源
- **FR-008**: 系统 MUST 在多组件已连接时保留所有合法组件的实时挂载状态，不因新增事件丢失既有端口映射

### Key Entities *(include if feature involves data)*

- **硬件插拔事件**: 表示一次插入、拔出、错误或全量快照变化，关键属性包括来源、时间、组件身份与接口身份
- **硬件连接快照**: 表示当前所有已连接组件与其接口位置的可信集合，用于驱动 readiness 判断与数字孪生投影
- **数字孪生场景**: 表示底座、接口定义以及当前挂载组件的显示结果，是前端渲染的直接输入

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在数字孪生首次打开后的 1 秒内看到仅包含底座模型的默认场景
- **SC-002**: 带有效接口身份的 mock 插入事件到达后，数字孪生在 1 秒内显示正确挂载结果
- **SC-003**: mock 拔出事件到达后，数字孪生在 1 秒内移除对应组件且不保留残影
- **SC-004**: 在包含两个及以上组件的快照场景中，系统能一次性显示全部合法组件，不丢失既有端口映射

## Assumptions

- mock 硬件插拔来源可能产出单组件事件，也可能产出全量快照
- 数字孪生页面继续由现有工作台承载，默认底座模型资源已经可用
- 组件是否“该显示”与“显示到哪个接口”都以 backend 归一化结果为准
- 本特性只解决端口同步与数字孪生实时更新，不扩展新的技能目录或工作流编排能力
