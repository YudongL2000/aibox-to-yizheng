# Feature Specification: OpenClaw 对话模式

**Feature Branch**: `001-openclaw-dialog-mode`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: "实现 OpenClaw 对话模式，覆盖极速响应、协作引导、智能共创三条交互分支，并让 Skill 匹配、硬件就绪校验、开始部署与教学模式跳转形成连续体验。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 已就绪技能的即时开玩 (Priority: P1)

作为已经接好硬件并进入对话模式的用户，我希望说出一个系统已掌握的技能后，AI 能立刻进入互动状态，而不是让我再做额外确认，这样我能获得“设备已经活过来”的即时反馈。

**Why this priority**: 这是对话模式最直接的价值兑现路径，决定用户第一次开口后会感受到“顺滑惊喜”还是“又一个流程入口”。

**Independent Test**: 仅准备一个已收录技能和一套完整硬件，让用户在对话模式说出技能请求；若系统能直接回应、展示玩法引导并伴随设备初始化动作，则该故事可独立验收。

**Acceptance Scenarios**:

1. **Given** 用户已进入对话模式、目标技能已存在且所需硬件均处于就绪状态，**When** 用户说出“跟我玩石头剪刀布”之类的已知技能请求，**Then** AI 必须在同一轮对话中给出充满期待感的确认话术，并补充足够的玩法引导，避免用户因忘记操作而停住。
2. **Given** 已知技能与所需硬件都已就绪，**When** AI 输出确认话术，**Then** 设备必须同步执行初始化动作，并且系统不得要求用户再点击“开始部署”或其他确认按钮才进入交互。
3. **Given** 用户处于对话模式并触发了已知技能，**When** 交互正式开始，**Then** 对话面板必须保持在当前会话上下文中，而不是切走到教学流程或重新发起搜索。

---

### User Story 2 - 缺件时的协作引导 (Priority: P2)

作为想立即体验某个已知技能、但当前硬件没有接全的用户，我希望系统能明确告诉我缺什么、插哪里、何时校验中、何时准备好，这样我能边插拔边完成配置，而不是被技术状态吓住。

**Why this priority**: 对话模式真正落地到物理世界时，缺件和插错口是高频场景；如果这条链不顺，用户会觉得系统“会聊天但不会落地”。

**Independent Test**: 仅准备一个已知技能并故意拔掉其中一个关键硬件，然后在对话模式发起请求；若系统能引导补插、在插入时显示校验反馈、成功后要求一次“开始部署”确认，则该故事可独立验收。

**Acceptance Scenarios**:

1. **Given** 用户请求一个已知技能，但至少一个关键硬件未连接，**When** 系统完成需求匹配，**Then** AI 必须明确说明缺失的硬件名称，并给出面向普通用户的插拔引导话术。
2. **Given** AI 已提示用户补插硬件，**When** 系统感知到用户插入新的硬件，**Then** UI 必须立刻进入“校验中”状态并展示简洁加载反馈，同时 AI 在校验完成前保持静默，不重复打断用户。
3. **Given** 校验成功且硬件与技能需求匹配，**When** 系统确认设备同步完成，**Then** AI 必须说明“已经准备好”，并且 UI 必须展示一个显著的“开始部署”动作入口。
4. **Given** 用户看到“开始部署”入口，**When** 用户点击该入口，**Then** 设备必须给出“苏醒”或准备完成的物理反馈，并开始正式互动。
5. **Given** 用户插错接口、插入错误型号或硬件异常，**When** 校验失败，**Then** AI 必须温和说明当前问题并鼓励用户继续检查，而不是直接结束会话或静默失败。

### User Story 3 - 未知技能的教学接力 (Priority: P3)

作为想让系统学会新本领的用户，当我说出一个未收录的目标时，我希望系统直接邀请我进入教学模式，并把我的原始意图完整带过去，这样我不用重新搜索或重打一遍需求。

**Why this priority**: 这条故事决定对话模式是不是一个“死路入口”。如果未知请求不能顺滑转教学，用户只会把它当成失败提示框。

**Independent Test**: 仅准备一个未收录技能请求，在对话模式中触发；若系统能直接出现“开启教学模式”入口，并在进入教学模式后带入原始学习目标，则该故事可独立验收。

**Acceptance Scenarios**:

1. **Given** 用户在对话模式中提出一个未被技能库收录的新目标，**When** 系统确认无法匹配现有技能，**Then** AI 必须以学习邀请的方式回应，而不是仅仅说“不支持”。
2. **Given** 系统确认当前目标是未知技能，**When** AI 给出回应，**Then** 对话流中必须同时出现一个可直接点击的“开启教学模式”动作入口。
3. **Given** 用户点击“开启教学模式”，**When** 页面跳转到教学工作流，**Then** 系统必须自动带入“学习该目标”的上下文，不要求用户重新输入刚才的请求。

### Edge Cases

- 当一个用户请求同时命中多个相近技能时，系统应优先选择最贴近当前可用硬件组合的技能，并向用户说明将执行的具体玩法名称。
- 当硬件在“校验中”或“准备就绪待部署”阶段又被拔出时，系统应立即撤销“开始部署”入口并回到缺件引导状态。
- 当硬件通道暂时不可用、无状态回传或同步超时时，系统应说明“当前还不能确认设备状态”，而不是误报“已准备好”。
- 当用户在校验尚未结束时重复发送相同请求，系统应复用当前校验过程，避免堆叠多个并发校验提示。
- 当用户已经进入教学模式后返回对话模式，系统应保留最近一次学习目标，便于继续接力而不丢失上下文。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供“对话模式”入口，让用户能在普通教学工作流之外，以对话方式调用现有技能。
- **FR-002**: 系统 MUST 在用户发出请求后，基于当前技能库判断该请求是否命中已知技能。
- **FR-003**: 系统 MUST 在技能命中后，基于最新硬件状态判断当前交互所需设备是否全部就绪。
- **FR-004**: 当技能命中且硬件全部就绪时，系统 MUST 直接进入互动准备状态，并在同一轮对话中输出确认话术与玩法引导。
- **FR-005**: 当技能命中且硬件全部就绪时，系统 MUST 触发与该技能相匹配的初始化物理反馈，并且不得要求用户额外点击部署按钮才开始互动。
- **FR-006**: 当技能命中但存在缺失硬件时，系统 MUST 明确指出缺失硬件名称，并以协作式语气指导用户完成插拔。
- **FR-007**: 当系统感知到新的硬件插入事件时，系统 MUST 立即向用户展示“校验中/同步中”的可见反馈，并在校验完成前暂停新的鼓励性播报。
- **FR-008**: 当缺失硬件校验成功后，系统 MUST 明确告诉用户“已准备好”，并展示显著的“开始部署”动作入口。
- **FR-009**: 当用户点击“开始部署”后，系统 MUST 触发一次代表“苏醒/就绪”的物理反馈，并进入正式互动状态。
- **FR-010**: 当硬件校验失败、插错接口或检测到异常时，系统 MUST 以温和且可操作的方式提示用户继续检查，而不是让会话静默终止。
- **FR-011**: 当用户提出未知技能请求时，系统 MUST 明确说明该技能尚未收录，并主动邀请用户进入教学模式。
- **FR-012**: 当用户点击“开启教学模式”后，系统 MUST 将原始对话目标无损带入教学模式，避免用户重新输入需求。
- **FR-013**: 系统 MUST 在对话模式界面中展示当前可直接唤醒的技能与其就绪情况，帮助用户理解“现在能玩什么”。
- **FR-014**: 系统 MUST 维持单一会话上下文，使技能判断、硬件校验、部署确认和教学模式跳转都围绕同一个用户请求展开。
- **FR-015**: 系统 MUST 记录每次对话模式请求的最终分支结果，至少区分为“直接开始互动”“缺件待部署”“跳转教学”与“校验失败待重试”四类结果。

### Key Entities *(include if feature involves data)*

- **Dialogue Mode Session**: 一次从用户发起技能请求到开始互动、等待补件、跳转教学或失败重试的完整会话，包含用户意图、当前分支和最近一次系统反馈。
- **Skill Readiness Snapshot**: 某个技能在当前时刻的可唤醒状态，至少包含技能名称、匹配结果、所需硬件清单和是否可直接开始。
- **Hardware Readiness Snapshot**: 当前设备集合的状态快照，至少包含已连接硬件、缺失硬件、最近一次插拔变化和最近一次校验结果。
- **Deployment Prompt**: 仅在“硬件补齐后待确认”路径下出现的动作状态，表示系统已经准备好、正在等待用户点击“开始部署”。
- **Teaching Handoff**: 从对话模式转入教学模式时携带的上下文对象，包含原始用户目标、建议学习文案和跳转时机。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对于已知技能且硬件齐全的请求，90% 的有效请求能在 3 秒内进入可直接互动状态，且不需要额外点击确认按钮。
- **SC-002**: 对于已知技能但缺件的请求，用户在插入正确硬件后 1 秒内能看到“校验中”反馈，且 90% 的成功校验能在 5 秒内给出“开始部署”入口。
- **SC-003**: 对于未知技能请求，95% 的用户能在一次点击内进入教学模式，并看到已自动带入的学习目标，无需重复输入。
- **SC-004**: 在首次体验测试中，85% 以上的用户能仅根据 AI 话术和界面提示判断自己当前处于“可直接开始”“需要补件”“可以去教学”三种状态中的哪一种。
- **SC-005**: 在硬件插错或校验失败的场景中，90% 的失败会话都能给出明确的下一步提示，而不是停留在无反馈或模糊状态。

## Assumptions

- 对话模式建立在现有 AI 助手面板之上，用户可以在同一界面中切换教学模式和对话模式。
- 系统已经存在一条可用的本地设备状态通道，能够持续回传已连接硬件、插拔变化和校验结果。
- 技能库中至少包含“石头剪刀布”这类可直接演示的技能，且每个技能都能声明自身所需硬件。
- 教学模式已经具备接收预填学习目标的能力，本特性只要求无缝转入，不重新设计教学画布本身。
