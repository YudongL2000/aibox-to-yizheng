# 2026-04-15

## New Facts

- 数字孪生场景的 `position/rotation` 解算链路本身没有漂移，`DigitalTwinMountResolver` 仍然输出预览窗口全局坐标系中的绝对位姿。
- 这次 3D 资产“坐标对了但模型还是偏”的真实根因在 `frontend/web/model_viewer/viewer.js`：viewer 一度停止把 GLB 平移到包围盒中心，转而直接使用 GLB 原始 pivot 作为 host 坐标原点。
- 当外部 `3dassets` 的内部 pivot 与几何中心或安装预期不一致时，保留原始 pivot 会让模型整体偏离接口锚点，即使 Flutter/scene 层传下来的世界坐标是正确的。
- 旧版 Tesseract viewer 语义是“wrapper 原点 = 模型包围盒中心”；恢复 `model.position.sub(center)` 后，外部资产的落点重新与 host 坐标对齐。

## Decisions

- 数字孪生 viewer 对外暴露的位姿锚点固定取模型包围盒中心，不能再把 GLB 原始 pivot 直接当成 host 坐标原点。
- 后续如果确实需要保留资产原始 pivot，必须显式引入单独的资产安装锚点字段或 asset-level compensation，不能静默改写 viewer 坐标语义。

## Constraints

- viewer 层的坐标锚点语义必须与 Flutter scene DTO、mount resolver、坐标编辑面板保持一致；任何一层改基准点都要同步更新整条链路。
- 外部 3D 资产的 pivot 不可控，前端渲染层不能假设所有 GLB 都以正确安装点建模。

## Retain

- 排查数字孪生“模型偏移”问题时，先分离“解算错”与“渲染锚点错”；如果 scene 数据正确，下一步优先检查 viewer 是按包围盒中心还是按原始 pivot 落点。
- viewer host 坐标一旦从包围盒中心切到 GLB 原始 pivot，所有外部资产都会把自身建模习惯泄漏进场景语义，坐标编辑会立刻失真。
- 想支持资产级差异，正确做法是增加显式 compensation/mount profile，而不是偷偷改变 viewer 的基础坐标原点。
