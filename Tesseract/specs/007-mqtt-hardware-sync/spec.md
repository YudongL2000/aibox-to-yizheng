# Feature Specification: MQTT Hardware Sync

**Feature Branch**: `007-mqtt-hardware-sync`  
**Created**: 2026-04-05  
**Status**: Draft  
**Input**: User description: "基于 docs/dev 中的 MQTT 与 P2P 示例，用真实 heartbeat/云端下发替换现有热插拔 mock 逻辑，统一驱动技能匹配、对话/教学模式、数字孪生显示、底部日志窗口、工作流上传/停止、麦克风/扬声器/摄像头预览，并将数字孪生接口数量调整为真实硬件 3-1.2/3-1.3/3-1.4/3-1.6/3-1.7 映射的 port_1~4 与 port_7。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时硬件连接与数字孪生同步 (Priority: P1)

作为正在教学模式或对话模式中操作真实硬件的用户，我希望客户端能根据真实硬件心跳与命令回包实时显示“硬件是否在线、哪个模块插在哪个接口、数字孪生是否已经挂载成功”，这样我不需要再依赖 mock 按钮猜测当前硬件状态。

**Why this priority**: 这是所有真实交互的前提。如果硬件连接状态、接口位置和数字孪生显示不是同一真相源，后续的技能唤醒、部署与预览都不可信。

**Independent Test**: 只连接硬件心跳流，不触发工作流上传，也能独立验证硬件状态框、底部日志、数字孪生接口挂载与顶部内置按钮是否随真实状态变化。

**Acceptance Scenarios**:

1. **Given** 客户端已启动且尚未收到硬件心跳，**When** 用户查看顶部硬件状态框与数字孪生窗口，**Then** 顶部显示“未连接硬件”，底部日志持续等待硬件心跳，数字孪生只显示底座与内置麦克风/扬声器按钮。
2. **Given** 云侧收到了真实硬件心跳，**When** 心跳中包含 `cam`、`hand`、`screen`、`car`、`wifi`、`mimiclaw` 等组件及其接口和在线状态，**Then** 客户端顶部状态框立即切换为已连接，底部日志持续追加心跳内容，数字孪生只把有模型的组件挂载到真实接口位置，`wifi` 与 `mimiclaw` 不渲染为伪模型。
3. **Given** 同一个组件的在线状态从 `online` 变为 `offline`，**When** 新心跳到达，**Then** 数字孪生移除该组件挂载，底部日志记录离线事件，并且对话/教学流程所见的硬件状态同步变更。

### User Story 2 - 完成配置后下发与控制真实硬件 (Priority: P2)

作为已经完成教学模式或对话模式配置的用户，我希望在“所有节点配置完毕”后能直接把工作流上传到真实硬件、停止工作流，并能从数字孪生顶部控制内置麦克风和扬声器，这样我能在同一个工作台里完成部署与基础验机。

**Why this priority**: 只有状态展示没有真实控制，用户依然要跳出系统使用调试脚本。将上传、停止、麦克风、扬声器控制拉回工作台，才能形成真正的部署闭环。

**Independent Test**: 在不使用摄像头预览的前提下，完成一次工作流配置后即可独立测试“上传工作流/停止工作流/打开麦克风/播放示例音频”及其日志和反馈。

**Acceptance Scenarios**:

1. **Given** 教学模式或对话模式已经进入“所有节点配置完毕”，**When** 用户点击“上传到硬件”，**Then** 客户端向硬件发送工作流下发指令、底部日志显示下发记录和回包结果，并在失败时给出明确原因。
2. **Given** 工作流正在硬件侧运行，**When** 用户点击“停止工作流”，**Then** 客户端发送停止指令，底部日志显示停止结果，界面反馈当前工作流已停止。
3. **Given** 数字孪生窗口已打开，**When** 用户点击顶部“麦克风”按钮，**Then** 客户端向硬件发送打开麦克风的指令，并在数字孪生左侧显示麦克风预览窗口与活动波形。
4. **Given** 数字孪生窗口已打开，**When** 用户点击顶部“扬声器”按钮，**Then** 客户端向硬件发送播放示例音频的指令，并在数字孪生左侧显示扬声器预览窗口与活动波形。

### User Story 3 - 技能命中后沿真实硬件链路进入分支 A-C (Priority: P3)

作为在对话模式中通过自然语言表达需求的用户，我希望系统先基于现有 Skills 库语义判断是否已有技能，再根据真实硬件在线状态决定是直接开始交互、等待补齐硬件，还是进入教学模式，而不是继续依赖前端硬编码卡片或 mock 选项。

**Why this priority**: 这是从“演示型原型”升级为“真实可用交互”的关键。对话模式必须服从技能库与真实硬件状态，否则分支 A-C 只是伪流程。

**Independent Test**: 在 Skills 库中预置一个可匹配技能，并接入真实心跳，即可独立测试“技能命中 + 硬件齐全/缺失/无技能”三条路径，无需验证摄像头预览。

**Acceptance Scenarios**:

1. **Given** Skills 库中已有与用户表达语义匹配的技能，且心跳显示所需硬件全部在线，**When** 用户在对话模式中提出该需求，**Then** 系统直接进入分支 A，显示技能引导话术，触发初始化物理动作，并允许直接开始交互。
2. **Given** Skills 库中已有与用户表达语义匹配的技能，但心跳显示仍缺少所需硬件，**When** 用户提出该需求，**Then** 系统进入分支 B，明确指出缺失硬件，等待真实心跳补齐后再显示部署按钮，而不是继续展示 mock 插拔入口。
3. **Given** Skills 库中找不到语义匹配技能，**When** 用户提出一个新的目标能力，**Then** 系统进入分支 C，提示进入教学模式，并把目标平滑带入教学工作流而不重复搜索或重复输入。

### Edge Cases

- 如果硬件心跳长时间中断，系统如何将“已连接”恢复为“未连接”，并撤销依赖该硬件的就绪态？
- 如果云侧命令已发送但端侧回包超时或失败，系统如何在日志、按钮状态和数字孪生上给出一致反馈？
- 如果同一时刻连续收到多条心跳，且包含不同组件的上线/下线变化，系统如何保证数字孪生最终只展示最后一份状态？
- 如果用户点击没有独立模型的组件（如 `wifi`、`mimiclaw`），系统如何避免显示错误模型，同时仍保留状态与日志可见性？
- 如果摄像头预览连接失败、流地址不可用或远端视频中断，系统如何给出可恢复的预览失败提示？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 以真实硬件心跳与命令回包作为教学模式、对话模式、数字孪生与日志窗口共享的唯一硬件真相源，不得再以前端 mock 插拔状态作为最终依据。
- **FR-002**: 系统 MUST 将真实接口数量调整为五个物理接口，并将 `3-1.2`、`3-1.3`、`3-1.4`、`3-1.6`、`3-1.7` 分别映射到 `port_1`、`port_2`、`port_3`、`port_4`、`port_7`。
- **FR-003**: 系统 MUST 在顶部“我的库”按钮左侧显示硬件连接状态框，并以最近一次有效硬件心跳是否到达作为“已连接/未连接”的判断依据。
- **FR-004**: 系统 MUST 将持续收到的硬件心跳与云侧下发到硬件的命令日志同步显示在客户端底部日志窗口中，并保持与真实事件顺序一致。
- **FR-005**: 系统 MUST 能从硬件心跳中解析组件名称、接口位置与在线状态，并据此准确更新数字孪生中的组件挂载或移除。
- **FR-006**: 系统 MUST 只为 `cam`、`hand`、`car`、`screen` 这些有独立模型的组件渲染数字孪生模型，并为 `wifi`、`mimiclaw` 保留状态与日志可见性但不渲染伪模型。
- **FR-007**: 系统 MUST 在数字孪生底座顶部提供“麦克风”和“扬声器”两个内置控制按钮，并允许在真实硬件未插外设时仍显示这两个按钮。
- **FR-008**: 系统 MUST 在教学模式或对话模式进入“所有节点配置完毕”后显示“上传工作流到硬件”与“停止工作流”两个控制按钮。
- **FR-009**: 系统 MUST 在用户点击“上传工作流到硬件”时向硬件发送工作流下发指令，并在用户点击“停止工作流”时发送停止当前工作流的指令。
- **FR-010**: 系统 MUST 在用户点击数字孪生顶部“麦克风”按钮时向硬件发送打开麦克风的指令，并在数字孪生左侧显示麦克风预览窗口与可见的音量波形反馈。
- **FR-011**: 系统 MUST 在用户点击数字孪生顶部“扬声器”按钮时向硬件发送播放示例音频的指令，并在数字孪生左侧显示扬声器预览窗口与可见的播放波形反馈。
- **FR-012**: 系统 MUST 在用户点击数字孪生中的摄像头模型时自动建立摄像头预览连接，并在数字孪生左侧显示当前摄像头画面。
- **FR-013**: 系统 MUST 在对话模式命中已有技能时，先基于现有 Skills 库和真实硬件状态判断分支 A/B/C，而不是继续依赖硬编码示例技能、前端 mock 技能卡或人工指定流程。
- **FR-014**: 系统 MUST 在分支 B 中等待真实硬件状态补齐后再进入“开始部署”，不得继续展示旧的 mock 插拔按钮作为唯一前进方式。
- **FR-015**: 系统 MUST 在分支 A 中，当技能已存在且所需硬件已就绪时，直接进入可交互状态，并伴随硬件初始化动作反馈。
- **FR-016**: 系统 MUST 在分支 C 中，当技能库中不存在语义匹配结果时，提示用户进入教学模式，并把用户目标直接带入教学入口。
- **FR-017**: 系统 MUST 在任何硬件心跳中断、命令失败、预览失败或工作流下发失败时，同时更新顶部状态框、底部日志、对话反馈与数字孪生状态，避免不同区域出现互相矛盾的信息。

### Key Entities *(include if feature involves data)*

- **Hardware Runtime Snapshot**: 表示最近一次真实硬件状态快照，包含连接时间、在线状态、组件清单、真实接口映射、以及哪些组件拥有可渲染模型。
- **Hardware Command Record**: 表示一次从云侧下发到硬件的控制动作及其回包结果，包含命令类型、目标组件、请求标识、发送时间、结果状态与可展示日志文本。
- **Skill Match Result**: 表示对话模式对用户输入与 Skills 库语义匹配后的结果，包含命中技能、缺失硬件、对应分支、以及是否允许直接交互或需要进入教学。
- **Preview Session**: 表示数字孪生左侧某种预览窗口的当前状态，包含预览类型、连接状态、最后活动时间、错误信息与用户可见反馈。
- **Hardware Workflow Deployment**: 表示一份待上传到硬件的工作流与其运行状态，包含来源工作流、用户填写参数、上传状态、停止状态与最近一次设备响应。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在收到有效硬件心跳后的 2 秒内，顶部状态框、底部日志和数字孪生至少有一个用户可见变化，并且变化内容彼此一致。
- **SC-002**: 对于有独立模型的硬件组件，95% 的插入或移除事件都能在单次状态变化后正确映射到对应接口，而不会显示在错误接口或保留残影。
- **SC-003**: 当用户在“所有节点配置完毕”后点击“上传工作流到硬件”或“停止工作流”时，用户能在 3 秒内看到明确的成功、失败或等待反馈。
- **SC-004**: 当用户点击麦克风、扬声器或摄像头入口后，用户能在 5 秒内看到对应预览区域进入“运行中”或“失败”状态，而不是无反馈等待。
- **SC-005**: 在对话模式中，用户提出一个已有技能需求时，系统能在一次回答中正确走入分支 A 或 B；提出未知技能需求时，系统能在一次回答中正确走入分支 C。

## Assumptions

- 真实硬件心跳会以稳定频率持续到达，系统可以根据最近一次心跳的时间窗口判断连接状态。
- 摄像头预览所需的流地址、会话参数或播放入口已经在现有示例或运行时配置中可获得，不需要用户额外手工填写复杂连接参数。
- 麦克风与扬声器预览对用户的主要价值是“确认已经进入运行状态”，因此可用可视化活动反馈来表达当前工作状态。
- 工作流上传与停止都由同一个硬件控制通道接收，且该通道能够回传可读的执行结果或失败信息。
- `wifi` 与 `mimiclaw` 虽然没有独立模型，但它们仍然是有效的硬件状态实体，需要出现在日志和系统状态判断中。
