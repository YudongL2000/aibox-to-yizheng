AI Agent 嵌入式需求澄清评估集 v1
1. 最终目标端到端评估工作流
- 理解用户需求
- 判断需求理解是否正确
- 对需求进行流程拆解
- 配置关键参数
- 生成端到端代码
- 最终烧录到端侧硬件，让硬件跑起来
2. 先落地“需求澄清阶段”评估集
- 把 case 正确分为以下四类：
  - A 类：需求清晰且在能力范围内，可直接进入下一阶段
  - B 类：需求清晰但超出能力范围，应拒绝
  - C 类：需求不清晰，需要继续澄清
  - D 类：多轮对话场景，需要结合上下文判断当前是否已闭环
- 核心考察以下能力：
  - 能力边界识别率：Agent 是否知道哪些需求当前产品生态支持，哪些不支持。
  - 必答点覆盖率：对于需要澄清的 case，Agent 是否问到了真正关键的问题，而不是泛泛乱问。
  - 追问克制度：对于已经足够明确的需求，Agent 是否避免了多余追问。
  - 上下文闭环判断能力：对于多轮对话，Agent 是否能在合适的时候停止追问并宣布需求已确认。
  - 幻觉控制：Agent 是否会捏造产品不支持的硬件、接口、算法能力或默认不存在的前提。
A 类：需求清晰且在能力范围内，可直接进入下一阶段
Case A-001
- 用户输入：我想做一个能自动跟着我走的小车。
- 预期判断：direct_accept
- 判断理由：目标明确，且可映射为摄像头 + 底盘小车 + 视觉跟随。
- 不应追问：不要追问是否需要摄像头这种显而易见的问题。
- 预期下一步动作：确认需求并进入方案拆解。
- Pass 标准：Agent 明确表示可做，并正确识别核心硬件组合。

Case A-002
- 用户输入：我说一句话，它就把我说的话重复出来。
- 预期判断：direct_accept
- 判断理由：目标明确，可映射为麦克风 + 扬声器。
- 不应追问：不要追问是否需要录音功能，因为需求已包含录音与播报。
- 预期下一步动作：确认需求并进入方案拆解。
- Pass 标准：Agent 能直接接受，不做无意义追问。

Case A-003
- 用户输入：做一个看到人就挥手打招呼的机器人，打招呼方式就是挥手。
- 预期判断：direct_accept
- 判断理由：检测对象明确，交互形式明确，可映射为摄像头 + 机械手。
- 不应追问：不要再问是语音还是挥手，因为用户已经说明。
- 预期下一步动作：确认需求并进入方案拆解。
- Pass 标准：Agent 正确识别摄像头 + 机械手，并可直接进入下一步。

Case A-004
- 用户输入：我想做一个机器人，看到我以后在屏幕上显示你好。
- 预期判断：direct_accept
- 判断理由：输入、触发条件、输出形式都已明确，可映射为摄像头 + 屏幕。
- 不应追问：不要追问通过什么方式打招呼。
- 预期下一步动作：确认需求并进入方案拆解。
- Pass 标准：Agent 接受需求，且不做多余澄清。

B 类：需求清晰但超出能力范围，应拒绝
Case B-001
- 用户输入：我想做一个自动浇花器，发现土干了就帮我浇水。
- 预期判断：reject_out_of_scope
- 判断理由：需要湿度传感器和水泵，但当前硬件清单不支持。
- 禁止幻觉：不能假装系统支持湿度检测或水泵控制。
- 预期下一步动作：礼貌拒绝，并说明是因为外设能力不支持。
- Pass 标准：Agent 明确指出超出边界，且拒绝理由准确。
Case B-002
- 用户输入：做一个会飞的无人机，能自动跟着我拍视频。
- 预期判断：reject_out_of_scope
- 判断理由：当前只支持底盘小车，不支持飞行模块。
- 禁止幻觉：不能把底盘小车硬解释成飞行器。
- 预期下一步动作：拒绝，并指出当前仅支持地面移动类能力。
- Pass 标准：Agent 能识别飞行超出能力边界。
Case B-003
- 用户输入：我想做一个房间太热就自动开空调的机器人。
- 预期判断：reject_out_of_scope
- 判断理由：需要温度传感器和红外发射能力，当前均不在支持范围。
- 禁止幻觉：不能编造可通过摄像头估温之类能力。
- 预期下一步动作：拒绝并解释缺失硬件。
- Pass 标准：拒绝清晰，理由贴合产品边界。
Case B-004
- 用户输入：让这个机器人闻到有人做饭就自动跑去厨房。
- 预期判断：reject_out_of_scope
- 判断理由：需要气味传感器，当前不支持。
- 禁止幻觉：不能编造麦克风可以闻味道。
- 预期下一步动作：拒绝并说明缺失感知模块。
- Pass 标准：Agent 不乱扩展能力，不胡编解决方案。
C 类：需求不清晰，需要继续澄清
Case C-001
- 用户输入：我想做一个看见我就能和我打招呼的机器人。
- 预期判断：clarify_needed
- 判断理由：输入条件有了，但输出形式不明确。
- 必答点：是语音打招呼、屏幕显示，还是机械手挥手。
- 禁止行为：不要直接默认一种打招呼方式。
- 预期下一步动作：继续追问交互方式。
- Pass 标准：追问精准，只问关键缺口。
Case C-002
- 用户输入：我想做一个能够和我玩石头剪刀布的机器人。
- 预期判断：clarify_needed
- 判断理由：输入方式与输出方式都不明确。
- 必答点：用户出拳是通过摄像头识别，还是别的输入方式；机器人出拳是屏幕显示，还是机械手动作。
- 禁止行为：不能默认摄像头识别 + 机械手输出。
- 预期下一步动作：追问交互闭环方案。
- Pass 标准：至少问到输入与输出两个核心点。
Case C-003
- 用户输入：帮我做一个能陪我聊天的机器人。
- 预期判断：clarify_needed
- 判断理由：聊天入口和输出方式不明确，是语音聊天、屏幕文字，还是两者都有。
- 必答点：你希望通过麦克风说话吗；机器人是用扬声器回复，还是屏幕显示回复。
- 禁止行为：不要默认支持完整大模型对话能力。
- 预期下一步动作：确认交互方式与能力预期。
- Pass 标准：问题聚焦交互方式与实现边界。
Case C-004
- 用户输入：我想做一个帮我看家的机器人。
- 预期判断：clarify_needed
- 判断理由：看家目标过于抽象，不清楚是巡逻、监控、提醒还是互动。
- 必答点：是固定监控还是底盘巡逻；发现异常后是语音提醒、屏幕提示还是别的反馈。
- 禁止行为：不要直接假设看家等于摄像头录像。
- 预期下一步动作：追问监控方式与反馈方式。
- Pass 标准：能把抽象目标拆成可执行澄清问题。
D 类：多轮对话，需要结合上下文判断是否闭环
Case D-001
用户第 1 轮说：我想做一个看见我就能和我打招呼的机器人。
用户第 2 轮说：我想让它直接开口说欢迎回来。
预期判断：context_complete。
判断理由：检测条件与输出形式都已经明确，可映射为摄像头 + 扬声器。
禁止行为：不要继续追问用户想怎么打招呼。
预期下一步动作：确认需求闭环并进入方案拆解。
Pass 标准：Agent 能及时停止追问。
Case D-002
用户第 1 轮说：我想做一个石头剪刀布机器人。
用户第 2 轮说：让它看我的手势，然后它用手跟我比划。
预期判断：context_still_incomplete。
判断理由：输入输出已明确，但是否要调用本地识别仍需确认。
必答点：你希望机器人真正识别你的手势，还是随机出拳。
预期下一步动作：继续追问识别方式。
Pass 标准：Agent 不会过早宣布信息闭环。
Case D-003
用户第 1 轮说：我想做一个监控家里宠物的小机器人。
用户第 2 轮说：让它在家里到处转着看。
用户第 3 轮说：看到宠物以后就在屏幕上显示“我找到你啦”。
预期判断：context_complete。
判断理由：移动方式与反馈方式都已明确，可映射为底盘小车 + 摄像头 + 屏幕。
预期下一步动作：进入方案拆解。
Pass 标准：Agent 能整合上下文，判断信息已足够。
Case D-004
用户第 1 轮说：我想做一个会聊天的机器人。
用户第 2 轮说：我对着它说话，它就回答我。
预期判断：context_still_incomplete。
判断理由：输入输出链路大致明确，但回答机制仍不明确，且存在能力边界风险。
必答点：你希望是固定话术回复，还是更复杂的智能对话。
预期下一步动作：进一步澄清能力预期。
Pass 标准：Agent 知道何时需要补一轮边界确认。

评分规则：
1. 决策正确性：decision 是否与 ground truth 一致。
2. 必答点召回率：对需要澄清的 case，是否覆盖核心追问点。
3. 冗余追问惩罚：对 A 类和已闭环 D 类 case，是否出现多余追问。
4. 幻觉惩罚：是否捏造当前硬件支持范围外的能力。
5. 硬件映射合理性：是否把用户目标映射到合理的硬件组合。
