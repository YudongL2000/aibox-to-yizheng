# Feature Specification: 工作流视图同步闭环

**Feature Branch**: `004-workflow-view-sync`  
**Created**: 2026-04-02  
**Status**: Draft  
**Input**: User description: "拉起客户端时应显示占位提示而不是 n8n 主页；点击“创建工作流”后，左侧客户端主工作区必须自动切换到刚创建的工作流，不能要求用户再手动点击“打开工作流”或依赖外部浏览器，同时需要参考现有 webUI 的自动切换体验并跑通 spec-kit 闭环。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建后即时看到工作流 (Priority: P1)

作为正在教学模式中生成流程的用户，我在聊天区点击“创建工作流”后，希望左侧活动工作区立即切换到刚创建的工作流画布，而不是继续停在主页、旧流程或等待我手动再点一次“打开工作流”。

**Why this priority**: 这是“教学模式生成流程”能否成立的核心闭环。只要这一步断掉，用户就会怀疑创建失败，整个体验等同于没完成。

**Independent Test**: 在客户端内触发创建工作流，若聊天区显示创建成功，则无需任何额外点击，左侧活动工作区应在同一客户端窗口内自动显示对应的新工作流画布。

**Acceptance Scenarios**:

1. **Given** 用户当前活动工作区还没有目标工作流，**When** 聊天区“创建工作流”动作返回成功，**Then** 左侧活动工作区必须自动切换到本次创建的工作流画布。
2. **Given** 创建动作已成功，**When** 客户端收到新的工作流引用，**Then** 用户不需要手动点击“打开工作流”也能在当前客户端看到结果。
3. **Given** 创建动作已成功，**When** 自动切换完成，**Then** 左侧活动工作区显示的必须是本次创建的目标工作流，而不是主页、旧流程或第三方流程。

---

### User Story 2 - 无工作流时显示明确占位态 (Priority: P2)

作为刚打开客户端或刚进入一个尚未生成流程的工作区的用户，我希望左侧活动工作区先显示一段清晰的占位提示，告诉我生成工作流后这里才会出现画布，而不是直接看到 n8n 主页或其他无关流程列表。

**Why this priority**: 占位态是判断“还没生成”还是“显示错了”的唯一视觉边界。没有它，用户无法区分系统未完成与系统串台。

**Independent Test**: 启动客户端或进入一个没有工作流引用的工作区时，左侧活动工作区应只显示占位提示文案，不出现主页、工作流列表或其他历史流程。

**Acceptance Scenarios**:

1. **Given** 用户打开客户端且当前活动工作区还没有工作流引用，**When** 左侧工作区初始化完成，**Then** 页面必须显示占位文案，说明生成工作流后这里会更新。
2. **Given** 当前活动工作区还没有工作流引用，**When** 工作区首次加载，**Then** 不得展示 n8n 主页、工作流列表或任何历史流程内容。

---

### User Story 3 - 异步准备阶段也能最终对齐 (Priority: P3)

作为连续操作的用户，我希望即使聊天区先拿到“创建成功”反馈，而左侧工作区还在加载或重连，客户端也能自己完成后续对齐，最终显示正确流程，而不是停留在错误页面。

**Why this priority**: 真实环境里聊天响应与嵌入式工作区准备速度常常不同步。如果没有这条恢复路径，系统会在边界时刻重新暴露主页或错误流程。

**Independent Test**: 在工作区尚未完全准备好时触发工作流创建，只要最终创建成功，主工作区仍应在稍后自动对齐到正确流程，无需重新输入或刷新浏览器。

**Acceptance Scenarios**:

1. **Given** 工作区仍在初始化或重连中，**When** 聊天区先收到工作流创建成功结果，**Then** 客户端应在工作区可用后自动完成切换。
2. **Given** 用户在自动切换前切换到另一个项目，**When** 之前项目的创建结果晚到，**Then** 该结果不得污染当前项目的主工作区。

### Edge Cases

- 当工作流创建成功，但主工作区尚未准备好渲染时，系统如何保证稍后仍会自动对齐？
- 当用户打开的是已有工作流引用的工作区，但当前引用已经失效或不可访问时，系统如何避免回退到无关主页？
- 当用户在创建流程的过程中切换工作区、切换项目或关闭当前视图时，如何防止旧结果更新到错误位置？
- 当后端成功返回多个工作流相关字段时，系统如何确保只认定当前项目对应的那一条引用？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在当前活动工作区没有已生成工作流引用时，显示明确的占位提示文案。
- **FR-002**: 系统 MUST 在无工作流引用的活动工作区中隐藏 n8n 主页、工作流列表和任何历史流程内容。
- **FR-003**: 当聊天区“创建工作流”动作成功后，系统 MUST 自动把当前客户端中的活动工作区切换到新创建的工作流。
- **FR-004**: 用户 MUST 能在当前客户端窗口中直接看到新创建的工作流，而不需要额外点击“打开工作流”。
- **FR-005**: 系统 MUST 将后端返回的工作流引用视为当前活动工作区的唯一事实来源。
- **FR-006**: 当工作区尚未准备好时，系统 MUST 在准备完成后继续自动对齐到已创建的目标工作流。
- **FR-007**: 系统 MUST 防止无关主页、历史流程或第三方流程占据一个尚未生成工作流的活动工作区。
- **FR-008**: 当用户切换工作区或切换项目时，系统 MUST 确保延迟到达的工作流创建结果不会更新到错误的工作区。
- **FR-009**: 如果工作流创建失败或缺少可用引用，系统 MUST 保持占位态并向用户显示可理解的失败反馈。
- **FR-010**: 外部浏览器打开工作流 MAY 作为辅助能力保留，但 MUST NOT 成为完成主流程闭环的必要步骤。

### Key Entities *(include if feature involves data)*

- **活动工作区**: 用户当前正在查看的单个客户端主画布上下文，拥有明确的显示边界，并负责承载占位态或目标工作流。
- **工作流引用**: 标识“当前活动工作区应显示哪一个流程”的业务引用，代表当前工作区与其目标工作流之间的绑定关系。
- **嵌入式工作区状态**: 描述主工作区当前处于占位态、等待对齐态或已显示目标流程态的界面状态。
- **创建会话结果**: 聊天区一次“创建工作流”动作产出的业务结果，用于驱动主工作区从占位态过渡到目标流程态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的“客户端启动且当前无工作流引用”场景在 2 秒内显示占位提示，而不是主页或历史流程内容。
- **SC-002**: 95% 的“创建工作流成功”场景在 5 秒内让主工作区自动显示新创建的流程，无需用户额外点击。
- **SC-003**: 在验收样本中，0 个“无工作流项目”场景允许无关主页、历史流程或第三方流程占据主工作区。
- **SC-004**: 在可用性走查中，至少 90% 的用户能在首次尝试中完成“创建并查看工作流”闭环，而无需借助外部浏览器。

## Assumptions

- 当前桌面客户端一次只聚焦一个活动工作区。
- 后端在工作流创建成功时能够提供足以唯一标识目标流程的引用信息。
- 现有“打开工作流”入口会继续保留为辅助动作，但它不应成为主路径依赖。
- 本轮范围只覆盖“客户端左侧活动工作区如何正确显示目标流程”，不扩展 workflow 编辑功能本身。
