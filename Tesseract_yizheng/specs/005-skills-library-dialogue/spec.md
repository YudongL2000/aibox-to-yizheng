# Feature Specification: 教学模式存库与对话模式真 Skill 分流

**Feature Branch**: `005-skills-library-dialogue`  
**Created**: 2026-04-02  
**Status**: Draft  
**Input**: User description: "先目前子仓根仓所有变更 commit 的部分，然后实现教学模式在用户拼接完所有组件后出现是否存入 skills 库的逻辑；用户确认后以 JSON 存入 skills 库，并带有该 skills 飞入 skills 库的 UI 动效；同时实现对话模式通过语义查询判断用户需求是否在现有 skills 库中存在，移除当前所有 mock skills 数据、对话模式 mock 硬编码选项逻辑与显示，并按 specs001 定义的分支 A-C 逻辑运行。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 教学完成后保存为技能 (Priority: P1)

作为刚完成一次教学装配的用户，我希望系统在所有组件拼接与工作流生成完成后，明确询问我要不要把这次成果保存进技能库；如果我确认保存，我能立刻看到“技能已收入我的库”的反馈，而不是让这次教学结果悄悄消失。

**Why this priority**: 这是技能库从“原型展示”变成“真实资产”的第一步。没有存库闭环，后续对话模式就没有可信的技能真相源。

**Independent Test**: 完成一次教学模式工作流生成后，系统弹出“是否存入技能库”确认；点击确认后，技能以 JSON 形式进入技能库，并在界面上看到明显的入库反馈与动效。

**Acceptance Scenarios**:

1. **Given** 用户在教学模式中完成了组件拼接并得到可部署的工作流，**When** 系统判定教学流程完成，**Then** 对话区必须出现“是否存入技能库”的明确确认动作，且默认不会自动保存。
2. **Given** 用户点击确认存入，**When** 保存成功，**Then** 系统必须显示技能飞入“我的库”的动效，并让“我的库”视图可见该技能。
3. **Given** 用户点击取消存入，**When** 当前教学完成，**Then** 工作流仍保留为本次教学成果，但技能库中不得新增记录。

---

### User Story 2 - 对话模式按真实技能库走 A/B/C 分支 (Priority: P1)

作为在对话模式中提出需求的用户，我希望系统先判断我的话是否命中了现有技能库，再决定是直接开玩、引导我插拔硬件，还是引导我进入教学模式，而不是只对少数写死的示例技能生效。

**Why this priority**: 对话模式是用户的主入口。如果分支判断还依赖 mock 技能和硬编码按钮，整个产品体验就是假的。

**Independent Test**: 预置至少一个真实保存进技能库的技能后，在对话模式输入“跟我玩石头剪刀布”与“帮我给花浇水”这两类请求，系统分别走已知技能分支和教学分支；不存在任何硬编码快捷技能选项。

**Acceptance Scenarios**:

1. **Given** 技能库中存在与用户语义匹配的技能，且所需硬件已就绪，**When** 用户在对话模式提出该需求，**Then** 系统必须直接进入 specs001 的分支 A，给出玩法引导并触发对应物理反馈。
2. **Given** 技能库中存在匹配技能，但所需硬件未就绪，**When** 用户提出该需求，**Then** 系统必须进入 specs001 的分支 B，引导插拔、校验通过后提供“开始部署”动作。
3. **Given** 技能库中不存在语义匹配技能，**When** 用户提出一个新的明确能力需求，**Then** 系统必须进入 specs001 的分支 C，提供“开启教学模式”动作，并带着目标平滑跳转。

---

### User Story 3 - 技能库与对话界面不再展示 mock 技能 (Priority: P2)

作为调试和验收这个系统的人，我希望界面上看到的技能列表、快捷唤醒卡片和“我的库”内容都来自真实技能库，而不是写死的三张卡片或三个演示按钮，这样我才能判断功能是否真的打通。

**Why this priority**: 只有移除所有假数据，测试结果才可信，产品才能避免被演示数据误导。

**Independent Test**: 清空技能库后进入对话模式与“我的库”，界面不应再显示任何硬编码技能卡片；保存一个技能后，界面只显示该真实技能。

**Acceptance Scenarios**:

1. **Given** 技能库为空，**When** 用户打开“我的库”或切换到对话模式，**Then** 系统必须显示真实空态，不得显示任何石头剪刀布/自动避障/添加技能等 mock 选项。
2. **Given** 技能库中有用户保存的技能，**When** 用户打开“我的库”，**Then** 只应显示真实保存的技能条目及其元数据。

### Edge Cases

- 用户保存的技能名称与现有技能高度相似时，系统必须避免重复生成多个语义上等价但名称略有差异的技能条目。
- 用户在保存动画进行中立即关闭技能库或切换视图时，保存结果仍必须可靠落库，且下次打开“我的库”能看到新技能。
- 用户在对话模式提出一个既像闲聊又像能力请求的模糊表达时，系统必须以 backend 语义判定为唯一真相源，不能由前端二次猜测。
- 技能库为空时，用户在对话模式输入普通闲聊仍应继续走 MimicLaw 兜底，而不是被误导去教学模式。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在教学模式完成组件拼接与工作流生成后，向用户明确提供“是否存入技能库”的确认动作。
- **FR-002**: 系统必须在用户确认存入后，将该技能以 JSON 形式持久化到统一的技能库中，并让后续会话可复用该技能。
- **FR-003**: 系统必须在保存成功后展示“技能飞入我的库”的可感知动效，并在技能库界面体现新增结果。
- **FR-004**: 系统必须允许用户拒绝存入技能库，且拒绝后不应产生任何新的技能库记录。
- **FR-005**: 系统必须让 backend 成为技能库检索与对话模式分流的唯一真相源，前端不得继续依赖 mock 技能数据或硬编码快捷技能逻辑。
- **FR-006**: 系统必须在对话模式接收到用户输入后，先基于技能库做语义查询，再决定进入 specs001 的分支 A、B、C 或 MimicLaw 兜底闲聊。
- **FR-007**: 系统必须移除当前对话模式界面中的所有 mock 技能按钮、硬编码快捷选项与假数据展示。
- **FR-008**: 系统必须让“我的库”界面只展示真实技能库内容；当技能库为空时，必须显示可信空态。
- **FR-009**: 系统必须让已保存技能包含足够的描述信息，以支撑对话模式的语义命中、硬件需求判断与后续玩法引导。
- **FR-010**: 系统必须让用户在对话模式命中已有技能后，按 specs001 的分支 A/B/C 规则获得一致的文本引导、部署动作与硬件校验反馈。

### Key Entities *(include if feature involves data)*

- **Saved Skill**: 用户通过教学模式生成并确认入库的技能资产，包含技能名称、语义描述、工作流定义摘要、硬件需求、玩法引导与可用于检索的关键词。
- **Skill Library**: 当前用户可复用的真实技能集合，支持持久化存储、按语义检索和在“我的库”中展示。
- **Skill Save Candidate**: 一次教学完成后待用户确认是否入库的临时技能快照，包含展示文案、目标技能信息与来源工作流上下文。
- **Dialogue Route Decision**: backend 根据用户输入与技能库内容做出的单一分流结论，决定进入分支 A、B、C 或 MimicLaw 兜底闲聊。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% 的教学完成会话都能在工作流生成结束后看到明确的“是否存入技能库”确认动作。
- **SC-002**: 用户确认保存后，新增技能在一次会话内可立即出现在“我的库”中，且刷新/重开后仍可见。
- **SC-003**: 当技能库为空时，对话模式与“我的库”界面不再显示任何硬编码技能卡片或快捷按钮。
- **SC-004**: 对于至少一个已保存技能与一个未知技能请求，系统分别能稳定进入 specs001 的已知技能分支和教学分支，且不依赖人工点击 mock 快捷项。
- **SC-005**: 对于普通闲聊输入，系统仍能把内容透传到 MimicLaw，并将回包显示到聊天记录中。

## Assumptions

- 当前阶段的技能库存储以本地 JSON 形式存在即可，不要求多人共享或云端同步。
- “飞入技能库”动效只需发生在 `aily-blockly` 客户端内，不要求跨窗口或跨进程动画同步。
- 技能库的主要消费方是 `aily-blockly` 对话模式与技能库界面；Flutter `frontend` 不属于这轮前端落点。
- 对话模式的 A/B/C 分支语义定义仍以 `specs/001-openclaw-dialog-mode` 为准，本 feature 只把分支判断从 mock 数据切到真实技能库。
